Model Prep: Handling Missing Values¶

This is a very common issue in working with data. Luckily tree-based models are the most resilient in handling them, due to their non-assumptive nature.

Key questions to ask yourself when handling missing values:

Is their occurrence random or not?

  • If they are random: You want to impute (ie, estimate) them
  • If they are not random: You want to encode them with their implied value

Non-Random Missing Values¶

This means the missing value has a specific meaning:

  • It never happened
  • 0
  • This thing doesn't exist
  • Absence, etc

In this case, it usually is best to fill them in with a single value that represents what that value means.

Denoting A Missing Value¶

Often if you have a dataset with a lot of missing values, it is helpful to note that a particular value was originally missing. It helps a model "see" whether or not something was there.

In tree based models, this allows them to make more meaningful splits higher up, allowing it to converge with a fewer number of boosting rounds.

In [5]:
# a useful function to help you with this
def denote_null_values(df):
    empty_cols_query = df.isnull().sum() > 0
    empty_df_cols = df.loc[:, empty_cols_query].columns.tolist()
    for col in empty_df_cols:
        col_name = f"{col}_missing"
        df[col_name] = pd.isnull(df[col])
    return df

Other Tip:¶

With tree based models, you can do the equivalent of denoting missing values just by filling them in with an extreme value that's out of range from everything else.

Values used are often -1 (if everything is positive), 999, -999, or something similar.

Most models do not allow you to do this, but since trees are insensitive to the scale of your data it gives you very similar results.

What About Randomly Missing Values?¶

If values are missing at random, that means their values won't consistently be one value or another. In this case, you need to impute them.

Some basic strategies:

  • mean for continuous numbers, mode for categorical values
  • you can use the results of a groupby to fill them in
  • you can use a model (like GradientBoosting!) to predict what they ought to be

Categorical Data¶

Lots of datasets have values that are represented as text.

Understanding what to do with these types of numbers is important, and today we're going to go over three different ways of doing this:

  • OneHotEncoding, or BinaryEncoding - Creating a 1, 0 representation for each unique value in a column
  • OrdinalEncoding - Randomly transforming each categorical value into a unique number

  • TargetEncoding - Using the average value of the target variable as the basis for each categorical value

    We'll discuss the general pros and cons of each of these methods, as well as items that are specific for tree-based models.

Before we do this -- let's install a useful library for encoding categorical variables:

category encoders: https://contrib.scikit-learn.org/category_encoders/

To use it, type the following into your command line: pip install category_encoders

Ordinal encoding assigns each unique value within a column an integer value. These can either be ordered (ie, Small -> 0, Medium -> 1, Large -> 2), or assigned at random.

The basic points behind this:

  • For practically all models this can be useful if you have categorical values that have a natural order to them
  • Outside of this instance, this is usually not a good idea for most models since it does not capture how each unique value might influence the target variable
  • Tree based models are uniquely well suited for ordinal data, and handle it surprisingly well -- this is because if given enough repetitions tree based models will eventually partition your dataset into its different chunks, regardless of how they're ordered
    • So in practice, this option is usually harmless for tree based models, but you might need to use additional boosting rounds to eventually separate the values into their naturally occurring clusterings

OneHot Encoding¶

  • Creates a binary 1, 0 representation for each unique value in a column
  • It does this by creating a separate column for each unique value within a column
In [18]:
from IPython.display import Image, display

display(Image(filename='assets/onehot_encoding.png'))

The basic points behind this:

  • This is probably the standard way of encoding categorical features for most models
  • Provides the most resolution with respect to how much each unique value aligns with the target variable

Bad Things:

  • This can add a lot of columns to your dataset. For some models this makes fitting either more time consuming or less reliable, and sometimes to the point that it will not converge at all
  • For variable values that don't occur frequently, this can make the values for this column less reliable, ie, if there's a categorical value that only occurs 30 times in a 250K row dataset, the numerical sparsity means it can be difficult to generate a useful statistical signal
  • For tree based models -- categorical columns with lots of unique values can make for very uneven splits, which can make it difficult for categorical values to get their proper due in model fitting

The Bottom Line¶

  • For tree based models, there is surprisingly little difference between these two
  • Somewhat paradoxically, ordinal encoding is often more reliable -- fitting times can be much faster if there are lots of unique values
  • Ordinal encoding will more likely give you lots of samples on each side of a split, which agrees with how trees fit their data

However.....

  • If you want insight into the impact of a particular value on its target, onehot encoding will give you more granular insight
  • For example, a feature importance score will not give unique values for different categories in an ordinal column
    • So if model insight is of the utmost importance, then one hot encoding can be more important

Common Problem: Categorical Columns That Have Lots of Unique Values¶

Very common issue is you have a column with categorical values that is very important, but has lots of unique values, making onehot encoding problematic.

Zip codes, medical billing codes, customer ids, etc.

The fancy term for this scenario is a high cardinality column

In our example, the restaurant id is a good example. It's probably quite important, but with 800+ unique values, it will be difficult to fit a model on onehot encoded variables for various reasons. Long fitting times, and tree based models will be slow to split on them, even if it's an important variable.

So, let's take a look at another technique that's useful for this scenario.

Target Encoding¶

A simple but effective way to deal with categorical columns that have lots of uniques (ie, high cardinality).

Very Basic Idea: You transform categorical values according to the average value of the target variable when that value exists.

Slightly Less Basic Idea: The value floats somewhere between the average value of y just for that category, vs. the average value for y as a whole.

That's all fine, but......

What about values that occur very infrequently?

Ie, what if we have a restaurant that has just opened, had a grand opening where they gave out free drinks to the ladies and had all main entrees for 50% off? The initial attendance values might be extremely high, but if we only had 7 or 8 days, there's a good chance this might not represent the normal or expected value for this restaurant under typical conditions.

Target encoding handles this in the following way.......

  • You can set a cutoff point that determines when you will use the average value for that particular category, and when you will use the average for the entire dataset.
  • In category_encoders this parameter is called min_samples_leaf

In this example, by setting min_samples_leaf to 21 (just more than the number of samples we had for this restaurant), the value began to move away from its true average, and much closer to the global average. If the value for min_samples_leaf begins to go much higher, the value will quickly get reduced to the global average.

This is a built in way to deal with overfitting -- it's simply difficult to conclude too much from something when there's not a lot of data. By changing this value, we can turn a knob to make sure we're not paying too much attention to values that have small sample sizes associated with them.

Pipelines¶

Your models will often require many preprocessing steps that need to be chained together. This can be problematic for things like making predictions on out of sample data, or separating operations for your training and test set.

Scikit-Learn has a pipeline module that makes it easy to encapsulate various processing steps into a single function to make your models more portable.

In [91]:
from sklearn.linear_model import LinearRegression
lreg = LinearRegression()
In [94]:
# let's go ahead and import it now
from sklearn.pipeline import make_pipeline

# let's initialize it with a target encoder + gbm
pipe = make_pipeline(ce.TargetEncoder(), lreg)

# and when we call score() -- it's applying all the processing steps in order to make the calculation

Important Points:

  • This is convenient. Less code. More work.
  • It makes it easy to deploy your model, and allow it to work with unseen samples
  • Makes data processing easier when you are performing operations on your training & test sets
